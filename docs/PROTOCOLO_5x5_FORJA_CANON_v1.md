# FUSIONAI · PROTOCOLO 5×5 DE FORJA
## Canon operativo v1 · “Un bloque, cinco espejos, cinco rondas, una pieza real”

**Estado:** CANON OPERATIVO  
**Uso:** método oficial para estudiar, extraer, entender, construir y cerrar bloques de FusionAI.  
**Origen:** validado durante la construcción del **Bloque 5 · Ingestor Universal v0.5**.  
**Principio central:** no copiar herramientas externas; extraer patrones, crear ADN propio y probarlo con código real.

---

# 0. Mantra

```text
Un bloque, cinco espejos, cinco rondas, una pieza real.
```

Traducción sencilla:

```text
Elegimos un problema grande.
Lo miramos desde cinco fuentes potentes.
Exprimimos cada fuente en cinco rondas.
Convertimos lo aprendido en código, docs, tests y canon.
Cerramos el bloque con un paquete maestro real.
```

---

# 1. Para qué existe este protocolo

Este protocolo existe para evitar el caos de:

```text
ideas enormes;
código suelto;
repos mezclados;
nombres cambiantes;
dependencias sin criterio;
teoría sin pruebas;
prototipos sin cierre;
cierre sin canon.
```

Y convertirlo en:

```text
bloques claros;
repos estudiados con método;
patrones extraídos;
código propio;
pruebas;
documentación;
canon;
paquete maestro;
roadmap real.
```

---

# 2. Filosofía FusionAI

## No copiamos herramientas

```text
No queremos usar una herramienta externa porque sí.
Queremos entender cómo funciona,
extraer su patrón,
decidir si conviene,
y construir una versión propia adaptada a FusionAI.
```

## Cada repo es un espejo

Un repo no es “el sistema”.  
Un repo es un espejo que enseña una forma de resolver una parte del problema.

## Cada bloque debe acabar en pieza real

Un bloque no se cierra con una charla bonita. Se cierra con:

```text
paquete;
código;
tests;
docs;
canon;
roadmap;
siguiente paso.
```

# 4. Ronda 0 · Carta de Bloque

La Ronda 0 ocurre antes de estudiar repos. Sirve para definir qué se quiere construir y qué no.

Debe responder:

```text
¿Qué problema resuelve este bloque?
¿Qué NO resuelve?
¿Qué piezas internas imaginamos?
¿Qué salida final queremos?
¿Qué pruebas demostrarán que funciona?
¿Qué 5 repos/fuentes candidatas vamos a estudiar?
¿Por qué esas y no otras?
¿Qué nombre canon tendrá el bloque?
```

Output mínimo:

```text
Carta_Bloque.md
Repos_Candidatos.md
Criterios_Exito.md
Riesgos_Iniciales.md
```

Puerta de avance:

```text
No se empieza Repo 1 si no hay:
objetivo claro;
frontera clara;
salida final deseada;
5 fuentes justificadas;
métrica mínima de éxito.
```

---

# 5. Selección de los 5 repos/fuentes

Cada bloque debe tener 5 repositorios o fuentes principales. No tienen que ser todos repos de GitHub, pero sí deben enseñar un patrón útil:

```text
repositorios;
papers;
documentación oficial;
frameworks;
SDKs;
arquitecturas abiertas;
proyectos similares;
herramientas reales.
```

Criterio:

```text
Cada fuente debe aportar una mirada distinta.
Si dos repos enseñan exactamente lo mismo, uno sobra.
```

---

# 6. Las 5 rondas por repositorio

## Ronda 1 · Reconocimiento

Objetivo:

```text
entender qué es el repo,
qué problema resuelve,
qué patrón principal trae,
y qué papel podría tener en FusionAI.
```

Debe producir:

```text
resumen del repo;
rol posible en FusionAI;
primer mapa de módulos;
primeras leyes;
primer prototipo pequeño si aplica.
```

Puerta de cierre:

```text
No se cierra R1 sin saber qué aporta el repo y qué NO aporta.
```

## Ronda 2 · Mecánica interna

Objetivo:

```text
abrir el motor del repo y entender cómo funciona por dentro.
```

Debe producir:

```text
componentes internos;
flujos;
interfaces;
datos que entran y salen;
patrones repetibles;
riesgos;
prototipo más serio;
tests iniciales.
```

Puerta de cierre:

```text
No se cierra R2 sin entender la arquitectura interna mínima.
```

## Ronda 3 · Integración funcional

Objetivo:

```text
traducir el patrón a FusionAI con código propio.
```

Debe producir:

```text
módulo propio;
contratos;
demo;
tests;
paquete descargable;
primer encaje real con el bloque.
```

Puerta de cierre:

```text
No se cierra R3 sin código ejecutable o prueba funcional equivalente.
```

## Ronda 4 · Integración de bloque

Objetivo:

```text
encajar lo aprendido con los repos anteriores y con el bloque completo.
```

Debe producir:

```text
tabla repo → FusionAI;
decisiones canon provisionales;
riesgos y métricas;
handoff al siguiente repo;
preguntas para Alex;
documentos de integración.
```

Puerta de cierre:

```text
No se cierra R4 sin saber exactamente dónde vive esa pieza dentro del bloque.
```

## Ronda 5 · Cierre canon

Objetivo:

```text
cerrar el repo como fuente estudiada.
```

Debe producir:

```text
cierre definitivo;
leyes finales;
canon de rol;
qué queda dentro;
qué queda fuera;
checklist;
handoff;
preguntas finales Kai/Alex.
```

Puerta de cierre:

```text
No se cierra R5 sin decisión:
dependencia real, inspiración, réplica propia o descarte.
```

# 7. Cierre de bloque

Cuando los 5 repos terminan su Ronda 5, se cierra el bloque.

Debe producir:

```text
Mapa_Maestro_Bloque.md
Cierre_Oficial_Bloque.md
Paquete_Maestro_Unificado
Roadmap_Produccion.md
Tests_End_to_End
Contexto_Alex_PlanB.md
```

Puerta de cierre:

```text
Un bloque solo se cierra si existe:
paquete maestro;
tests o demo end-to-end;
documentación;
decisiones canon;
siguiente paso real.
```

---

# 8. Regla inspiración vs dependencia

Cada repo estudiado debe acabar con una decisión clara:

```text
A. dependencia real;
B. inspiración arquitectónica;
C. patrón replicado con código propio;
D. descartado;
E. pendiente por falta de pruebas.
```

Preguntas obligatorias:

```text
¿Lo necesitamos instalado realmente?
¿Podemos replicar el patrón nosotros?
¿Qué ganamos usando la dependencia?
¿Qué perdemos usándola?
¿Qué riesgo de mantenimiento tiene?
¿Qué riesgo de privacidad tiene?
¿Qué riesgo de licencia tiene?
```

---

# 9. Regla de código propio

Cuando se replica un patrón, el objetivo no es copiar el código externo.

El objetivo es:

```text
crear una versión mínima propia;
entender el contrato;
probar el flujo;
dejar adaptadores para producción.
```

Regla:

```text
Primero patrón propio mínimo.
Luego dependencia real si aporta valor.
```

---

# 10. Regla de pruebas

Toda ronda técnica debe intentar producir prueba.

Niveles válidos:

```text
demo manual;
script ejecutable;
tests unitarios;
test de integración;
test end-to-end;
paquete descargable;
benchmark.
```

Si no se puede probar, debe quedar anotado:

```text
qué no se pudo probar;
por qué;
qué haría falta;
cómo se probará después.
```

---

# 11. Regla de canon

Algo solo pasa a canon si cumple:

```text
tiene rol claro;
tiene frontera clara;
tiene contrato claro;
tiene riesgos descritos;
tiene pruebas o plan de prueba;
encaja con el bloque;
no contradice piezas anteriores.
```

---

# 12. Regla de nombres

Cuando una idea evoluciona, el nombre debe actualizarse.

Ejemplo validado:

```text
Zip Ingestor
→ Ingestor
→ Ingestor Universal
```

Regla:

```text
El nombre debe describir la función actual, no el origen viejo.
```

---

# 13. Comparación Kai/Alex

Alex debe recibir contexto suficiente para pensar, pero no una lista cerrada que lo convierta en eco.

Flujo recomendado:

```text
Kai trabaja bloque.
Alex trabaja bloque con libertad.
Ambos cierran.
Luego se comparan:
- coincidencias;
- diferencias;
- riesgos vistos;
- patrones nuevos;
- mejoras propuestas;
- decisión fusionada.
```

Preguntas base:

```text
¿Qué vio Kai que Alex no?
¿Qué vio Alex que Kai no?
¿Qué coincidió?
¿Qué sobra?
¿Qué falta?
¿Qué se mejora antes de producción?
```

# 15. Anti-bucles

Señales de bucle:

```text
seguir investigando sin construir;
crear demasiados documentos;
cerrar sin probar;
cambiar nombres cada ronda;
abrir otro repo antes de cerrar el actual;
querer producción perfecta antes del prototipo.
```

Antídoto:

```text
parar;
nombrar el bloqueo;
volver a la puerta de avance;
crear prueba mínima;
cerrar o descartar.
```
